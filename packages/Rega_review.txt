# Rega #3822

## General

* The package is well-structured with a clear separation of concerns (API
  client generation, parsing, validation, workflows), which is commendable.
* Good use of modern packages like `httr2`, `validate`, and `rlang`.
* The custom error handling in the main submission workflow provides excellent
  robustness.

## DESCRIPTION

* Looks good.

## NAMESPACE

* Looks good.

## vignettes

* The vignette is clear and easy to follow.
* The API client generation functions (`create_client`, `extract_api`) are
  very general. Consider spinning this out into a separate, reusable package
  for building API clients from OpenAPI specifications.

## R

* In `apiclient.R`, the `parse_ega_body` function contains a developer
  comment: `in the datasets response, there are 2 columns with status, why?`.
  Please remove comment from production code.
* In `format.R`, the `process_delimited_column` function has hard-coded logic
  for `pubmed_ids`. To improve generability, consider making such
  transformations data-driven (e.g., passing a list of column-specific type
  conversions) rather than using `if/else` statements for specific columns.
* In `workflows.R`, the `new_submission` function is quite long. For better
  readability and maintainability, consider refactoring the major steps
  (submitting samples, studies, etc.) into smaller internal helper functions.
* The use of `Sys.getenv` and `askpass::askpass` in `access.R` for
  credential management is a good but consider using the `keyring` package for
  more secure and flexible credential storage.
* The manual creation of lookup tables (LUTs) in `parse.R` from API
  responses is a potential point of fragility if the API response structure
  changes. This is an inherent challenge, but it's worth noting.

## tests
