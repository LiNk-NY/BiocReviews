# scGraphVerse #3820

* It would be nice to have the package use more integrative Bioconductor data
  representations rather than primarily using `list` type objects. Have you
  considered using other data structures?
* The package seems to do too much in a single package, i.e., network
  inference, community detection, network comparison, and visualization. Have
  you considered splitting the package into multiple smaller packages with
  more focused functionality?

## DESCRIPTION

* The package seems to depend on quite a high number of other packages. Is
  there a way to reduce the number of dependencies? A package with many
  dependencies can be hard to maintain and install.
* Consider seperating out the plotting functions into a separate package, in
  order to reduce the number of dependencies for users who may not need the
  plotting functionality.

## vignettes

* Consider using a helper function to preprocess the data in the `loaddata`
  chunk, in order to avoid repeating the same code multiple times.
* There is a `list()` function call in the `consensus` chunk that seems to be
  superfluous and can it be removed?


## R

* Consider renaming the `adj_truth`, `count_matrices`, etc. datasets to
  something more descriptive, e.g., `toy_adj_matrix`, to improve code
  readability.
* Can you show how the `adj_truth` dataset is used in the examples? It is
  loaded in the `community_path` example but not used.
* Some functions such as `community_similarity` and `compare_consensus` do too
  much (i.e., computes community assignment, topological properties, and
  visualizes) for a single function. Consider breaking them down into smaller,
  more focused functions to improve readability and maintainability.
* Improve accuracy of error messages, e.g., in `compare_consensus` the
  error reads "binary adjacency matrix" but the function is only checking
  that the `consensus_matrix` input is a matrix.
* For functions with multiple argument options, e.g., `method` argument in
  `create_consensus` (and `cutoff_adjacency`), add all possible methods in the
  function signature, i.e., `c("vote", "union", "INet")` and use
  `match.arg(method)` in the function body to validate the input.
* The `earlyj` function does not seem to be filtering the `input_list`
  properly, the error message states that the input should be a list of
  matrices, seurat obj, or SCE obj, but the function only checks if the input
  is a list.
* Ensure that all functions return properly documented values, e.g., in the
  `edge_mining`, one of the return calls provides a simple `data.frame` but
  the documentation states that a "named list of data.frames" is returned.
* Remove the carrige returns in the error messages in `infer_networks`. You
  can use `\n` to insert a new line in the error message, e.g.,
  `stop("Python not available.\nIs it configured?", " Try again.")`
* Consider combining the multiple `message` calls in `init_py` into a single
  call.
* The package should not add methods to classes that are not defined in it,
  e.g., `PCzinb,SummarizedExperiment-method`. Only create methods for classes
  that are defined in the package. Use only a plain function that can adapt
  to different input classes instead.
* Use `seq_len()` instead of `seq(1, n)` (or `(i+1):(i+j)`) to avoid potential
  issues when n is 0 in `Poisk2()`.
* Consider using `BiocBaseUtils::checkInstalled()` in `plotg()`,
  `stringdb_adjacency`, etc. to check if the required R packages are
  installed, instead of relying on multiple `requireNamespace()` calls.
* (Optional) Consider using `httr2` instead of `httr` for improved HTTP
  requests handling in `stringdb_adjacency`.
* Remove empty `.onLoad` and `.onUnload` functions in `R/zzz.R`.
* Also, the `.onAttach` function is not needed and creates too much noise for
  the user. It is generally not a good practice to display a message every
  time the package is loaded. It is better to avoid using `onAttach` to reduce
  startup time and avoid cluttering the user's console with messages.
