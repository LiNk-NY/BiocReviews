# RNAmodR #1106

Overall, good work on the packages. There is a lot of code in the package that
implements checks for type or non-NULL. It would be good to have a device to
avoid such repeated checks. Perhaps using a particular data structure where
such `list` inputs are not used.

## DESCRIPTION

* Looks good.

## NAMESPACE

* A `settings` function seems a bit vague as to what it does. It would be
helpful for users to have a more descriptive name, if possible.
* Also, see `AnnotationHub` for a similar generic.

## vignettes

* It seems like the early part of the creation vignette goes more into the
developer rather than the user side of the package. You should indicate that
the vignette is for developers only.
* Please use examples that would be mainly used by the end user.
* Main vignette looks good and thorough.

## R/

* Signature arguments in `setGeneric` are not required as they match the formal
arguments unless you are using lazy evaluation.
* Did you contact the 'Gviz' maintainer about using
a non-hardcoded DNA alphabet?
* You shouldn't need to run `valid*` functions in the `show` method, this
should be done at class creation.
* Be more specific with error messages rather than 'Something went wrong.' If
it is an internal error, you can tag it with '<internal>'.
* Use default arguments such as `input = list()` instead of assigning after a
check for `missing()`.
* It may be useful to create a helper function to test `input[["x.measure"]]`
for non-zero, NA, and blanks instead of using a series of `if` and `else` calls,
especially when you run this check on several occasions.
* You may benefit from using a map to check arguments for `.norm_plot_args` and
others. This map will indicate (as separate columns) the error message, assert
function, and name in `input` to check for all those that exist.
* Avoid nesting `lapply`, where possible.
* There's a lot of code in the package and I wasn't able to review all of it.

Minor:
* Avoid name clashing objects with function names, e.g., `list <- ...`
* No need to wrap character/logical/numeric scalars with the `c()` function.
* Use `Map` to avoid `SIMPLIFY = FALSE` argument in `mapply`.
* No need to check `if(X == TRUE)`, simply do `if (X)`. The inverse also applies,
i.e., `if (!X)` is preferred over `if(X == FALSE)`
